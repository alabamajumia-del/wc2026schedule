# 08. 竞品参考优化方案

创建日期：2026-05-23

用途：把竞品和相邻优秀工具页的设计经验落地成 wc26schedule 后续页面优化标准。该文档用于指导后续开发，不直接作为公开页面内容。

核心判断：

```text
wc26schedule 不应该只做 SEO 内容站，而应该做 SEO + 工具型赛程站。
```

优秀竞品的共同点不是“文章写得长”，而是让用户能在页面里完成任务：筛选、切换视图、下载、同步日历、查看地图、追踪晋级路径。这样的页面更容易提升停留时长、复用率和回访率。

## 1. 参考对象总览

| 参考对象 | 页面类型 | 核心价值 | 适合借鉴到本站 |
| --- | --- | --- | --- |
| MatchTimes | 赛程筛选工具 | 时区、球队、分组、赛程筛选 | 主赛程页、国家时间页 |
| Match Time Calendar | 日历下载工具 | Google/Apple/Outlook 日历导入 | Calendar / ICS 页面 |
| Cup26Map | 地图型赛程页 | 城市、地图、比赛日探索 | Host Cities、城市页 |
| KickoffClock | 下载与时区工具 | PDF/Excel/CSV/JSON 下载说明 | PDF、Excel、下载中心 |
| WorldCupHub | 日期分组赛程 | 按日期卡片浏览，移动友好 | 主赛程页 date view |
| Wallchart.io | 互动赛程墙 | 积分、晋级、淘汰赛路径 | Bracket、Standings |
| SportsCal | 自动同步日历 | 订阅赛程、自动更新 | Calendar sync、团队/城市日历 |

## 2. 可复用设计原则

后续页面优化必须优先考虑以下原则：

1. 页面首屏必须直接表达“用户能做什么”，不能只是标题、介绍和事实卡。
2. 核心页必须有可操作元素，例如筛选、下载、切换、同步、地图或互动表。
3. 同一份赛程数据要拆成多种使用方式：表格、卡片、地图、PDF、Excel、CSV、ICS。
4. 移动端不能只依赖宽表格，必须逐步引入卡片化赛程视图。
5. 下载页必须像文件工具页，而不是普通文章页。
6. 城市页必须有地点感，不能只是城市名替换。
7. Bracket 和 Standings 必须逐步工具化，不能长期停留在解释性文章。
8. 页面 SEO 档案必须记录“参考对象”和“页面工具类型”。

## 3. 页面类型模型

后续每个页面优化前，需要先判断页面属于哪类模型。

| 模型 | 页面目标 | 关键模块 |
| --- | --- | --- |
| Schedule Tool | 查完整赛程并筛选 | 搜索、日期、球队、城市、阶段筛选；表格/卡片切换 |
| Download Tool | 下载并复用文件 | 文件预览、格式对比、下载按钮、更新时间、适用场景 |
| Calendar Tool | 把赛程放入用户日历 | 全部/球队/城市/阶段 ICS，Google/Apple/Outlook 指引 |
| City Explorer | 以城市规划比赛 | 地图、城市卡、球场、比赛日、交通/住宿提示 |
| Team Tracker | 跟踪球队路线 | 小组赛程、对手、城市、可能晋级路径、阵容更新 |
| Standings Tool | 跟踪积分和晋级 | 小组表、规则、赛果、晋级状态 |
| Bracket Tool | 追踪淘汰赛路径 | 可视化 bracket、占位路径、晋级联动 |
| Viewing Guide | 找观看方式 | 国家/地区、频道、streaming、时区转换 |
| Ticket Guide | 买票前决策 | 官方入口、阶段、风险提醒、城市/日期规划 |

## 4. 竞品拆解与落地方案

### 4.1 MatchTimes

参考价值：

- 页面围绕“查比赛时间”组织。
- 支持赛程、分组、球队、语言/地区入口。
- 更像工具，不像传统文章。

落地到 wc26schedule：

- 主赛程页必须保留并强化筛选工具。
- 增加按用户地区/时区查看的页面或组件。
- 后续新增国家时间页：

```text
/world-cup-2026-schedule-usa-time/
/world-cup-2026-schedule-uk-time/
/world-cup-2026-schedule-india-time/
/world-cup-2026-schedule-australia-time/
```

主赛程页优化方向：

- 首屏展示筛选器预览。
- 增加 Table / Date / Team / City 视图切换。
- 每个团队、城市、日期都能继续点击进入独立页。

### 4.2 Match Time Calendar

参考价值：

- 页面目标非常单一：下载或订阅日历。
- 用户完成操作后，赛程会进入自己的 Google / Apple / Outlook Calendar。
- 复用率高于 PDF，因为日历会持续被查看。

落地到 wc26schedule：

- 新建 Calendar 页面：

```text
/world-cup-2026-calendar/
```

功能优先级：

1. 全部比赛 `.ics` 下载。
2. 按球队 `.ics` 下载。
3. 按城市 `.ics` 下载。
4. 按阶段 `.ics` 下载。
5. 后续做订阅式 calendar feed。

页面结构建议：

```text
H1: World Cup 2026 Calendar
H2: Add the World Cup 2026 Schedule to Your Calendar
H2: Choose Calendar Type
H2: Download All Matches
H2: Team Calendar Downloads
H2: City Calendar Downloads
H2: Google, Apple and Outlook Instructions
H2: Update and Time Zone Notes
FAQ
```

### 4.3 Cup26Map

参考价值：

- 地图让城市页和 host city 内容更直观。
- 用户可以按城市探索比赛，而不只是读表格。
- 视觉主题强，赛事空间感清晰。

落地到 wc26schedule：

- `/world-cup-2026-host-cities/` 应从列表页升级为 City Explorer。
- 每个城市页需要更强的本地内容，而不是统一模板。

Host Cities 页应增加：

- 16 个城市的地图式或区域式入口。
- 城市卡片：比赛数、球场、日期范围、国家。
- 筛选：USA / Canada / Mexico，Group / Knockout，日期范围。
- 城市页链接。

城市页应增加：

- 球场图片或地图占位。
- 机场/交通/住宿区。
- 比赛日清单。
- “适合哪类球迷”说明。

### 4.4 KickoffClock

参考价值：

- 下载中心清晰解释 PDF、Excel、CSV、JSON 的不同用途。
- 强调时区转换和下载文件的实用性。

落地到 wc26schedule：

- PDF 页和 Excel 页必须继续工具化。
- 后续可以增加 JSON 下载，服务开发者和数据用户。

下载页文件卡应包含：

```text
Format
Best for
Included fields
Last updated
File status
Primary action
Fallback action
```

PDF 页当前落地状态：

- 已加入 PDF 文件预览。
- 已加入下载检查清单。
- 已加入 PDF / Excel / Live schedule 差异说明。

Excel 页下一步优化要求：

- 不要复用 PDF 页结构。
- 必须强调筛选、排序、工作簿 sheet、字段、旅行计划和自定义备注。

### 4.5 WorldCupHub

参考价值：

- 按日期组织赛程，移动端浏览更友好。
- 比 104 行大表更容易扫读。

落地到 wc26schedule：

- 主赛程页增加 Date View。
- 移动端优先显示 Match Cards。
- 桌面端保留完整表格。

主赛程页后续视图切换：

```text
Table View
Date View
Team View
City View
```

Match Card 字段：

```text
Match number
Date
Kickoff time
Teams
Stage / Group
City
Stadium
Team links
City link
```

### 4.6 Wallchart.io

参考价值：

- 互动墙能让用户预测、跟踪、复访。
- 积分、晋级和 bracket 联动是世界杯期间的高价值功能。

落地到 wc26schedule：

- `/world-cup-2026-standings/` 必须做成 Standings Tool。
- `/world-cup-2026-bracket/` 必须做成 Bracket Tool。

Standings 页后续功能：

- 48 队小组积分表。
- 胜平负、进球、失球、净胜球、积分。
- 晋级状态。
- 规则说明。

Bracket 页后续功能：

- Round of 32 到 Final 的路径图。
- 占位符阶段展示。
- 比赛开始后联动晋级队伍。
- 每场 knockout match 链接到城市和球队。

### 4.7 SportsCal

参考价值：

- 赛程同步到用户日历是最强复用机制。
- 自动更新能力带来长期留存。

落地到 wc26schedule：

- 先做静态 `.ics`。
- 后续再考虑订阅式 feed。

Calendar 页需要支持：

```text
All matches
One team
One city
Group stage only
Knockout stage only
Final only
```

## 6. 后续页面优化流程更新

每个页面优化前，必须在对应 `page-seo-plans/{slug}.md` 增加以下字段：

```text
Reference model:
Competitor inspiration:
Page tool type:

Retention mechanism:
Mobile behavior:
```

示例：

```text
Reference model: KickoffClock
Page tool type: Download Tool
Hero interaction: File preview + format checklist
Reusable action: PDF download, Excel fallback
Retention mechanism: Fresh copy before ticket/travel decisions
Mobile behavior: File preview stacks above download cards
```

## 7. 判断页面是否仍然模板化

优化后必须检查：

- 首屏是否能一眼看出页面独特任务。
- H2 是否不是通用标题。
- 是否有页面独有的交互或视觉模块。
- 是否有页面独有的使用路径。
- 是否不是只换关键词、换页面名。
- 是否能回答“用户为什么会回来”。

如果一个页面只能靠正文解释用途，而首屏和模块无法表达用途，则仍然视为模板化。
